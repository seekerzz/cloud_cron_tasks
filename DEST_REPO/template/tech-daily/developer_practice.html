1

Harness Engineering：AI Agent 时代的工程新范式

随着 AI Agent 能力快速提升，"Harness Engineering"（脚手架工程）正在成为区分普通开发者与顶尖开发者的关键技能。Anthropic 最新技术博客指出：当模型本身越来越强时，真正拉开差距的是围绕模型构建的整套运行系统。

这套系统包括任务拆解、上下文管理、多智能体协作、评估与反馈、QA 验证、结构化交接等组件。对于长时运行的复杂编码任务，单个 Agent 很容易随着时间推移逐渐失控——上下文越长，模型越容易失去连贯性，甚至表现出"上下文焦虑"（提前收尾、接近上下文极限就想结束工作）。

核心变化

工程师角色从"逐行实现"转向"迭代监督"，聚焦设计意图、系统级推理和质量把控。Cognition 联合创始人 Scott Wu 表示："我们 Cognition 的工程师现在已经不打代码了——这就是现实。管理 Devon 一小时，抵得上自己干六到十二小时。"

实践建议

1) 使用 context reset 而非简单压缩，给 Agent 真正的"干净起点"；2) 设计结构化 handoff artifact 进行阶段间状态交接；3) 建立评估闭环，用 outcome-based rewards 训练垂直 Agent 模型。参考 Kimi K2.5、Cursor Composer 2、Chroma Context-1 的技术报告，它们都采用了相同的底层配方：强基座模型 + 生产环境内训练 + 结果导向奖励。

AI 编码工具并未加快项目交付：编码从来都不是瓶颈

Agoda 软件工程师 Leonardo Stern 发表观察文章，指出尽管 AI 编码工具提高了个人开发者产出，但项目层面的整体开发速度提升远不如预期。Faros AI 研究分析了 1255 个团队、逾万名开发者的数据：高度采用 AI 的团队完成任务量增加 21%，合并 PR 增加 98%，但 PR 评审时间增加 91%。

这说明编码阶段加速会将压力转移到其他环节。开发瓶颈已向上游转移至需求定义与验证环节，因为这些领域更依赖人工判断。这是对 Fred Brooks "没有银弹"论断的重新印证——仅对开发生命周期单一环节提速，对整体交付效率的提升会逐渐递减。

关键洞察

传统上要求组建精简高效工程团队的逻辑建立在"编码是最具价值的活动，而沟通是制约效率的成本"这一假设上。如果创造最高价值的工作转变为协作式需求定义与架构对齐，那么沟通不再是要尽量减少的开销，沟通本身就是核心工作。小团队的优势不在于减少协调，而在于更快达成共识。

3

LLM Post-Training 全景：从 RLHF 到 Agentic RL

Meta FAIR 研究科学家 Haohe 发布《LLM Post-Training 全景指南》，系统梳理了从 2022 年 ChatGPT 至今的后训练技术演进。核心洞察：SFT 教模型"说什么"，偏好优化教模型"怎么选"，而 RL 教模型"怎么想"。

当前主流技术栈包括：1) SFT（监督微调）作为基础；2) RLHF / DPO / GRPO 进行偏好对齐；3) Agentic RL 让模型学会使用工具、多步推理。Kimi K2.5 通过 RL 学会生成并行子代理，Cursor 在生产 harness 中训练自总结能力，Chroma 的 20B 检索模型学会在搜索中自我剪枝上下文。

实践意义

对于构建垂直领域 Agent 的开发者，建议参考这一"强基座 + 生产环境训练 + 结果奖励"的配方。不要在通用模型上简单微调，而是要在真实业务场景中设计完整的训练闭环，让模型在可验证的任务目标上持续优化。

Netflix 图抽象架构：毫秒级处理 650TB 图数据

Netflix 工程师分享了其高吞吐图抽象系统的设计。该平台支撑多项内部业务：Netflix Gaming 社交图、服务拓扑图（用于故障排查和根因分析）。核心设计决策：将边连接关系与边属性分离存储，全球部署数据副本。

系统通过限制遍历深度、通常要求指定起始节点，以牺牲部分查询灵活性换取大规模场景下稳定的低延迟。最新图状态存储于键值抽象，历史变更通过时序抽象记录，与 EVCache 分布式缓存层深度集成。

可借鉴经验

对于需要处理复杂关联数据的系统，Netflix 的架构提供了重要参考：1) 区分实时查询和离线分析的不同存储需求；2) Schema 内存化强制数据校验并优化遍历路径；3) 时序抽象支持图演化分析和审计追溯。

Tool Calling 能力测试：27B 模型表现最优的启示

开发者 Stevibe 构建了 Tool Calling 测试框架，在 15 个场景、12 个工具上测试了 Qwen3.5 系列从 0.8B 到 397B 的所有尺寸。意外发现：27B 密集模型和蒸馏版 27B 全部通过测试，而 397B 失败 2 项、122B 失败 1 项、35B 失败 2 项。

最暴露问题的测试是"查冰岛人口然后计算 2%"——许多大模型直接用记忆里的近似值而非工具返回的数据，表现出对工具的不信任。小模型则容易陷入循环反复调用工具直到超时。

关键结论

小模型会幻觉数据，大模型会无视数据，27B 刚好把链路跑通了。这提示开发者：Agent 场景不一定需要最大参数模型，中等规模的专用模型在工具调用和指令遵循上可能表现更好。选择模型时应针对具体任务进行实测，而非盲目追求参数规模。

供应链安全新威胁：Unicode 私有字符隐藏恶意代码

Aikido Security 披露新型供应链攻击：攻击者向 GitHub 上传 151 个恶意软件包，利用 Unicode "私有使用区"（Private Use Areas）隐藏恶意载荷。这些字符对应美式英文字母表但完全不可见，代码审查人员或静态分析工具看来一片空白，而 JavaScript 解释器运行时会由解码程序还原并执行。

攻击波及 GitHub、NPM 和 Open VSX。解码后的载荷通过 Solana 区块链传输，拉取并执行第二阶段脚本窃取开发者资产。这种攻击方式对传统基于文本匹配的检测机制构成严峻挑战。

防御建议

1) 在 CI/CD 中增加 Unicode 异常字符检测；2) 使用支持 Unicode 规范化的代码审查工具；3) 对依赖包进行行为分析而非仅静态扫描；4) 考虑使用 lockfile 并审查每次依赖变更的实际内容。
